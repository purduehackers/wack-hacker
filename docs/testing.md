# Testing

Vitest with Istanbul coverage. The test setup deliberately splits unit and integration tests, and excludes the parts of the codebase that are either glue or thin wrappers around external services.

## Suites

| Command                    | Suite           | Config                               |
| -------------------------- | --------------- | ------------------------------------ |
| `bun run test`             | Unit            | `vitest.config.ts`                   |
| `bun run test:integration` | Integration     | `vitest.integration.config.ts`       |
| `bun run test:coverage`    | Unit + coverage | `vitest.config.ts` with `--coverage` |

Test files live next to the code they cover:

- Unit: `*.test.ts`
- Integration: `*.integration.test.ts`

The unit config explicitly excludes `*.integration.test.ts` so they don't run twice.

## Coverage thresholds

| Metric     | Threshold |
| ---------- | --------- |
| Statements | 90%       |
| Branches   | 85%       |
| Functions  | 90%       |
| Lines      | 90%       |

These thresholds apply to the **bot/AI core** — they are enforced after excluding the directories that are framework glue or external API wrappers.

### What's excluded from coverage

Set in `vitest.config.ts` under `coverage.exclude`:

| Path                                       | Why                                                          |
| ------------------------------------------ | ------------------------------------------------------------ |
| `src/app/**`                               | Next.js route shells — just call into `server/`              |
| `src/env.ts`                               | Validates env at boot; tested implicitly by every other test |
| `src/server/**`                            | Hono route wiring; integration tests cover this              |
| `src/workflows/**`                         | Workflow DevKit code; needs the runtime to test meaningfully |
| `src/bot/handlers/**`                      | Handler glue; bulk of the work is in shared helpers          |
| `src/bot/integrations/**`                  | External SDK wrappers                                        |
| `src/lib/ai/tools/**`                      | Tool implementations are thin shells around vendor SDKs      |
| `src/lib/ai/skills/generated/**`           | Generated by `compile-skills.ts`                             |
| `src/lib/protocol/utils.ts`                | Tiny helpers; covered transitively                           |
| `src/lib/ascii.ts`, `src/lib/test/**`      | ASCII art and shared test fixtures                           |
| `src/**/{types,constants,enums,errors}.ts` | No runtime behavior to cover                                 |

The intent is to enforce a high bar on the parts of the code where bugs hurt — the `EventRouter`, the `SkillRegistry`, `AgentContext`, dedup/locking, packet codec, cron parser, conversation store — and not waste effort on integration glue or third-party shells.

## Test fixtures

`src/lib/test/fixtures/` holds shared fixtures, including AI SDK mocks. Import from there rather than rebuilding mocks in each test file. The fixtures directory is excluded from coverage.

## Adding a test

1. Create `<module>.test.ts` next to the file you're testing.
2. Use the `@/` import alias (configured in `vitest.config.ts`) to import from `src/`.
3. For anything that touches an AI SDK call, use the mocks in `src/lib/test/fixtures/`.
4. For anything that touches Redis or an external service, write an integration test instead (`<module>.integration.test.ts`) and use the `vitest.integration.config.ts` runner.

Run a single file:

```bash
bun run test src/bot/router.test.ts
bun run test:integration src/bot/store.integration.test.ts
```
